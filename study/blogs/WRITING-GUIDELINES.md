# 学习博文写作规范

本规范用于后续每完成一段完整知识点课程后，沉淀对应博文草稿。

## 写作目标

每篇博文同时服务两个目标：

1. 强化作者对课程知识点的长期记忆。
2. 分享给想了解 Agent Runtime / Harness / 工程化 Agent 的读者。

## 默认读者

默认读者是：

```text
有一定后端 / AI 应用 / Agent 平台经验，但不一定读过 Pi 源码的人。
```

因此文章应：

- 能读懂，不堆源码。
- 有工程类比，不纯讲概念。
- 有关键流程，不写成流水账。
- 有个人理解，不只是复述文档。

## 重要保密要求

1. 不暴露作者所在企业的真实行业方向、业务系统、项目名称、客户、供应链、内部平台或内部流程。
2. 如需企业场景类比，默认使用“金融行业 Agent”作为泛化例子。
3. 金融行业例子只用于技术类比，不代表真实公司业务，不包含真实客户、真实账户、真实风控策略、真实交易数据。
4. 避免出现可识别的内部命名、业务线、组织结构、系统代号。

推荐使用的泛化例子：

```text
风控 Agent
信贷审批 Agent
交易审计 Agent
客服工单 Agent
合规审核 Agent
理财问答 Agent
反欺诈分析 Agent
```

不推荐使用：

```text
真实企业行业方向
真实系统名称
真实业务链路
真实客户字段
真实内部平台名称
真实接口名
```

## 文章数量规则

默认一课一篇。

如果某节课内容特别大，可以拆成：

```text
主概念篇
源码流程篇
企业设计篇
阶段总结篇
```

但拆分前必须说明拆分原因，避免为了发文数量而拆分。

## 推荐文章结构

每篇文章建议采用：

```text
1. 标题
2. 开篇问题：为什么这个知识点重要？
3. 一句话结论
4. 背景铺垫：读者需要知道什么
5. 核心概念解释
6. 流程图 / 分层图 / 对照表
7. 少量关键代码片段或伪代码
8. 工程类比：默认用金融行业 Agent
9. 常见误区
10. 我的阶段性理解
11. 小结
```

## 代码片段规范

可以放代码，但不能让读者感觉在读源码。

规则：

1. 每篇最多 2-4 个代码片段。
2. 优先放伪代码、简化结构、关键接口，不贴长源码。
3. 每个代码片段后必须解释“它说明了什么”。
4. 代码片段服务理解，不服务炫技。

推荐：

```ts
runAgentLoop(messages, context, config, emit, signal, streamFn)
```

不推荐：

```text
大段复制源码实现细节。
```

## 图表规范

高概念密度文章必须至少包含一种结构化表达：

```text
表格
ASCII 流程图
职责边界图
分层图
检查清单
```

必要时可后续生成更美观的图片版本。

## 风格要求

- 中文为主。
- 语气自然，像一篇学习笔记，不像官方文档。
- 尽量使用“我以前以为 / 现在理解”这种学习型表达。
- 概念解释要落到工程场景。
- 复杂内容先给结论，再拆细节。
- 每篇都要有一句“可记忆的核心句”。

## 禁止事项

- 不写成公司内部分享稿。
- 不暴露真实企业行业方向。
- 不贴大段源码。
- 不把文章写成 API 文档。
- 不为了完整而塞太多分支内容。
- 不未经理解就写“我已经掌握”。

## 文件命名规范

博客草稿放在：

```text
study/blogs/drafts/
```

命名格式：

```text
0001-agent-loop.md
0002-agent-message-vs-llm-message.md
0003-tool-call-lifecycle.md
0004-agent-harness-runtime-control.md
```

后续发布版可放在：

```text
study/blogs/published/
```

## 每次写博文后的记录

每次生成或更新博文后，应更新：

```text
study/blogs/README.md
study/README.md 中的博客模块说明
必要时更新 NOTES.md 中的写作偏好
```
