# 学习节奏设计

本文档记录本学习工作区的协作方式。它回答一个问题：后续课程应该如何推进，才能让 Agent Harness 学习有迹可循、可持续复盘，并且跨会话后不丢失讲课风格。

完整教学执行要求以：

```text
study/TEACHING-PROTOCOL.md
```

为准。

## 总体节奏

```text
课程设计
  -> 总览图 / 分层图
  -> 源码精读
  -> 概念解释
  -> 小练习
  -> 理解检查
  -> 反馈纠偏
  -> 复盘沉淀
  -> 学习记录
  -> 必要时博客草稿
```

## 每节课的固定结构

每一节课尽量控制在一个小主题内，结构如下：

1. **本课位置**：说明本课属于 MASTER-PLAN 的哪个阶段。
2. **本课目标**：本节课结束后应该能做到什么。
3. **源码范围**：只指定少量关键文件和函数。
4. **总览图**：先建立整体结构，不直接进入细节。
5. **核心概念表**：用表格把概念边界说清楚。
6. **源码路径**：按调用链读，而不是按文件从头到尾读。
7. **工程类比**：结合 Java / Dify / 企业 AI 平台经验。
8. **检查问题**：用 2-5 个问题验证是否理解。
9. **小练习**：尽量能在项目里实际操作或画出结构图。
10. **复盘结论**：总结本课最重要的工程判断。

## 高概念密度课程

对于 AgentHarness、Session、Compaction、Extension、Provider、Observability、企业级设计等课程，必须使用结构化讲解：

```text
总览图
分层表格
流程图或职责边界图
源码 / 资料锚点
金融行业 Agent 类比
常见误区
小结卡片
理解检查
```

不要一开始就堆源码、术语或长段解释。

## 课程与记录的关系

```text
course-design/   # 课程草案：准备教什么
lessons/         # 正式课程：用户可复习的学习材料
reference/       # 压缩后的速查资料：架构图、流程图、对照表
learning-records/# 学习记录：用户真正理解后的关键认知
blogs/           # 学完完整知识点后的博文草稿
```

## 什么情况下写学习记录

学习记录不是日报，不记录“今天看了什么”。只有满足以下情况才写：

- 用户能用自己的话解释一个非显然概念。
- 用户完成了一个能验证理解的小练习。
- 用户纠正了一个重要误解。
- 用户的学习目标发生了变化。

示例：

```text
0001-agent-loop-is-model-tool-observation-cycle.md
0002-tool-is-governed-execution-unit.md
0003-harness-is-runtime-control-layer.md
```

## 学习中的角色分工

### 老师负责

- 设计课程顺序。
- 指定源码阅读范围。
- 解释关键概念。
- 提供练习和验收标准。
- 根据反馈纠偏。
- 控制课程节奏，复杂课程分段讲。
- 在理解被验证后写学习记录。
- 每节课后更新 handoff / reference / progress / current。
- 完整知识点完成后，按需要生成博客草稿。

### 学习者负责

- 按课程阅读指定源码。
- 尝试用自己的话解释理解。
- 完成阶段练习。
- 提出困惑、反例和工程联想。
- 把 Pi 的设计和自己已有的 Java / Dify / 企业 AI 平台经验建立连接。

## 推荐单次学习节奏

```text
10 分钟：看课程目标和概念图
20 分钟：读指定源码
15 分钟：回答检查问题
30-60 分钟：完成小练习
10 分钟：复盘记录
```

实际节奏可以根据工作时间调整，但每次都应尽量完成一个闭环。

## 防止学习发散的规则

- 不因为看到新目录就立刻跳过去。
- 不在第一阶段深入所有 Provider、UI、包管理、发布流程。
- 不把 Pi 当作普通 CLI 项目读，而是围绕 Agent Harness 主线读。
- 不把 Dify 经验丢掉，而是持续做对照：Dify 平台能力在 Pi 里对应哪一层底层实现。
- 动态补课必须服务主计划，补课后必须回到主线。
