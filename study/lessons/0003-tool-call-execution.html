0003：Tool Call 执行链路

状态：已完成。对应参考资料：tool-call-lifecycle.md

本课目标

理解 Pi 中 tool call 从模型生成到工具执行、结果回灌的完整生命周期。

核心结论

LLM 只能提出 toolCall；
Agent Runtime 才真正决定工具能不能执行、怎么执行、结果如何治理和回灌。

主生命周期

assistant message contains toolCall
  -> executeToolCalls()
  -> choose sequential / parallel
  -> emit tool_execution_start
  -> prepareToolCall()
      -> find tool
      -> prepareArguments
      -> validateToolArguments
      -> beforeToolCall
  -> executePreparedToolCall()
      -> tool.execute(...)
      -> optional tool_execution_update
  -> finalizeExecutedToolCall()
      -> afterToolCall
  -> emit tool_execution_end
  -> createToolResultMessage()
  -> append to currentContext.messages
  -> next LLM turn

prepare 阶段

prepareToolCall 是工具执行前的安全关口。它负责找工具、修正参数、校验参数，并执行 beforeToolCall。

prepareArguments = 参数整理 / 兼容 / 修正
validateToolArguments = Schema 校验 / 正式参数校验

beforeToolCall 与 afterToolCall

  • beforeToolCall：执行前治理，适合权限、审批、风控、白名单、危险命令拦截。
  • afterToolCall：执行后治理，适合脱敏、压缩、错误标准化、审计补充、结果改写。

错误也要回灌

工具不存在、参数不合法、权限拒绝、执行异常，都应该变成 error toolResult message，而不是静默跳过或只打印日志。

role: "toolResult"
isError: true
content: [{ type: "text", text: "失败原因" }]
