0002：AgentMessage vs LLM Message

状态：已完成。对应参考资料：agent-message-vs-llm-message.md

本课目标

理解 Pi 为什么要区分内部消息模型和 LLM 消息模型，并掌握 convertToLlm 的边界意义。

核心结论

AgentMessage = Agent 内部完整消息账本
LLM Message  = LLM Provider 接口 DTO

AgentMessage 服务 Agent Runtime，LLM Message 服务模型接口。两者不能混为一谈。

为什么要分层

Agent 内部不仅有 user、assistant、toolResult，还可能有 UI 状态、审计事件、权限决策、业务状态、bash 执行记录、分支摘要和压缩摘要。

这些消息不一定都该进入模型，所以 Pi 使用 convertToLlm 作为边界转换器。

源码锚点

packages/agent/src/types.ts
packages/agent/src/agent.ts
packages/agent/src/agent-loop.ts
packages/agent/src/harness/messages.ts

基础层默认转换

defaultConvertToLlm(messages)
  -> keep user
  -> keep assistant
  -> keep toolResult
  -> filter others

基础 Agent 默认只让模型看到标准协议消息。

Harness 层扩展消息

  • bashExecution：记录命令、输出、exitCode、截断状态等。
  • custom：可显示或可转换的自定义消息。
  • branchSummary：分支返回后的摘要。
  • compactionSummary：历史压缩后的摘要。

四个判断维度

1. 是否写入 session？
2. 是否展示给用户？
3. 是否进入 LLM？
4. 是否影响工具调用或控制流？

企业 Agent 设计原则

  • 用户消息通常进入 LLM。
  • assistant 消息通常进入 LLM。
  • 工具观察结果通常进入 LLM，但要压缩或脱敏。
  • UI 状态默认不进入 LLM。
  • 审计和权限原始事件默认不进入 LLM。
  • 会影响模型下一步行为的业务结论，可以转换后进入 LLM。

本课检查结果

用户已经能解释 AgentMessage 与 LLM Message 的职责边界，能判断 branchSummary、compactionSummary、bashExecution、AuditEventMessage、ClearanceStatusMessage 是否以及如何进入 LLM。
