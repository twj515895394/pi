本课目标

读懂一次用户输入如何进入 Pi 的 Agent Loop，并理解它如何变成模型响应、工具调用、工具结果回灌和下一轮模型调用。

核心心智模型

messages -> LLM -> assistant message -> tool calls -> tool results -> messages -> LLM

Agent Loop 不是简单聊天逻辑，而是模型、工具、消息和事件共同组成的运行闭环。

主调用链

Agent.prompt(input)
  -> normalizePromptInput(input)
  -> runPromptMessages(messages)
  -> createContextSnapshot()
  -> createLoopConfig()
  -> runAgentLoop(prompts, context, config, emit, signal, streamFn)
  -> runLoop(currentContext, newMessages, config, signal, emit, streamFn)
  -> streamAssistantResponse(...)
  -> executeToolCalls(...)
  -> createToolResultMessage(...)
  -> next LLM turn or agent_end

关键概念

  • Agent：有状态封装，负责保存 transcript、处理事件、暴露 prompt / steer / followUp 等 API。
  • AgentLoop：底层运行循环，负责模型调用、工具调用、事件发射和下一轮判断。
  • AgentMessage：Pi 内部消息模型。
  • LLM Message：模型 API 能理解的消息格式。
  • ToolResult message：工具结果包装成消息后回灌到上下文，让模型继续推理。

transformContext 与 convertToLlm

transformContext 处理 AgentMessage 层面的上下文，例如压缩、裁剪、注入外部上下文。

convertToLlm 把 AgentMessage 转成 LLM 可理解的 Message，并过滤 UI-only / status 类消息。

AgentMessage[]
  -> transformContext()
  -> convertToLlm()
  -> Message[]
  -> LLM

工具执行生命周期

assistant message contains toolCall
  -> executeToolCalls()
  -> prepareToolCall()
  -> beforeToolCall()
  -> tool.execute()
  -> afterToolCall()
  -> createToolResultMessage()
  -> push back to context.messages
  -> next LLM turn

事件与状态更新

AgentLoop 不直接修改 Agent state，而是 emit 事件；Agent 通过 processEvents() 消费事件并更新状态。

  • message_start：消息开始。
  • message_update：assistant streaming 增量更新。
  • message_end：消息定稿并写入状态。
  • tool_execution_start：工具开始执行。
  • tool_execution_update：工具执行过程更新。
  • tool_execution_end：工具执行结束。
  • agent_end：本次 run 结束。

本课检查结果

用户已经能解释 AgentMessage 的统一输入、currentContext 与 newMessages 的区别、toolResult 回灌、transformContext 与 convertToLlm 的分工，以及事件驱动状态更新的设计意义。

已纠正两个易错点：UI-only status 应在 convertToLlm 阶段过滤；工具事件名应使用 tool_execution_start/update/end。
